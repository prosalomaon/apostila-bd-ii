Teoria & Conceitos

A DDL (Data Definition Language) é o subconjunto da SQL que permite ao projetista definir a estrutura, organização e as relações entre os dados armazenados. Nesta etapa de implementação física, transformamos atributos em colunas e definimos seus Datatypes (tipos de dados) e Domínios.

Classificação de Tipos de Dados (ANSI/SQL)

Os tipos de dados mais utilizados em SQL são divididos em categorias principais de acordo com a natureza da informação:

1. Dados Numéricos

  • Integer (ou Int): Inteiros padrão (4 bytes). Recomendado para Chaves Primárias e Estrangeiras devido à alta performance em Joins.
  • Smallint e Tinyint: Versões de 2 e 1 byte para economia extrema de espaço.
  • Decimal (ou Numeric): Números exatos com precisão e escala definidas. Essencial para valores financeiros.
  • Float e Real: Números aproximados com ponto flutuante. Evitar quando a precisão absoluta é exigida.
  • Money / Smallmoney: Tipos especializados para transações comerciais que garantem precisão monetária.

2. Dados Alfanuméricos (Strings)

  • Char(n): Comprimento fixo. Preenche com espaços se o texto for menor que n. Ideal para campos de tamanho constante (ex: siglas de estados).
  • Varchar(n): Comprimento variável. Utiliza apenas o espaço necessário. Recomendado para nomes e descrições.
  • Text / Long Varchar: Usados para grandes volumes de texto livre (observações, memos).

3. Dados Temporais (Data e Hora)

  • Date: Apenas a data (CCYY-MM-DD).
  • Time: Apenas o horário (hh:mm:ss).
  • Datetime: Combina data e hora. Amplo suporte (ano 1000 a 9999).
  • Timestamp: Data e hora com fuso horário (UTC). Frequentemente usado para auditoria (limite até o ano 2038).

4. Dados Binários e Outros

  • Binary / Varbinary: Dados brutos (arquivos compilados).
  • Blob / Image: Grandes objetos binários (fotos, PDFs, músicas).
  • Bit: Valores 0 ou 1 (campos booleanos/verdadeiro-falso).

Recomendações Gerais

  • Atomicidade: Cada coluna deve armazenar valores indivisíveis.
  • Compatibilidade: Chaves Estrangeiras devem possuir exatamente o mesmo tipo de dado da Primária referenciada.
  • Nulidade: Use NOT NULL para colunas essenciais à existência do registro.
  • Uso de Índices: Indexe colunas frequentemente usadas em filtros (WHERE) ou junções.

Exemplo de Código

Abaixo, um script de criação de tabela aplicando diferentes tipos de dados e restrições de integridade baseadas em regras de negócio:

-- Criando a tabela de Produtos com restrições de domínio
CREATE TABLE PRODUTO (
    codigo_produto  INT NOT NULL,           -- Inteiro para melhor performance
    descricao       VARCHAR(50) NOT NULL,   -- Alfanumérico de tamanho variável
    unidade         CHAR(3) DEFAULT 'UN',   -- Tamanho fixo com valor padrão
    preco_unitario  MONEY NOT NULL,         -- Tipo monetário para precisão
    tipo_produto    VARCHAR(20) NULL,
    
    -- Restrição de Chave Primária
    CONSTRAINT PK_Produtos PRIMARY KEY (codigo_produto),
    
    -- Regra de Validação: Preço não pode ser zero ou negativo
    CONSTRAINT CH_Preco_Positivo CHECK (preco_unitario > 0)
);

Dicas & Lembretes

  • Regra de Integridade de Identidade: Nenhum elemento que componha uma Chave Primária (PK) pode aceitar valores nulos, pois isso impediria a identificação única da tupla.
  • VARCHAR vs CHAR: Use CHAR para dados de tamanho fixo (ex: siglas de estados ou unidades) e VARCHAR para nomes ou descrições, economizando espaço em disco.
  • Integridade Referencial: Ao definir uma Chave Estrangeira (FK), os atributos devem possuir o mesmo domínio da Chave Primária referenciada.

Como aplicar na Prática

  • Analise o Minimundo: Identifique se os dados são numéricos, textuais ou temporais.
  • Defina a Nulidade: Questione-se: "Este dado é essencial para a existência do registro?". Se sim, use NOT NULL.
  • Implemente Regras de Negócio: Use a cláusula CHECK para impedir dados inconsistentes, como datas de nascimento no futuro ou salários negativos.
