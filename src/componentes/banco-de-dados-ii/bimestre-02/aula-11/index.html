Teoria & Conceitos

Os predicados são filtros lógicos que determinam a "sobrevivência" de um registro no conjunto de resultados, funcionando como a camada de inteligência da cláusula WHERE.

1. LIKE (Busca de Padrões)

Diferente do operador de igualdade (=), o LIKE trabalha com aproximação e padrões de texto.

  • Cenário Básico: "Quero todos os nomes que começam com A". → LIKE 'A%'
  • Cenário Avançado: "Quero códigos de produto que seguem o padrão: duas letras, qualquer coisa no meio e terminam com o número 5". → LIKE 'AB%5'
  • O Curinga _ (Underline): Frequentemente negligenciado, ele é vital para padrões de tamanho fixo. Ex: LIKE '___-2024' encontra qualquer código de 3 caracteres que seja do ano de 2024.

Diferenças de Implementação: MySQL vs. PostgreSQL

Recurso MySQL PostgreSQL
Sensibilidade (Case) Geralmente Case-Insensitive (depende do Collation da coluna). Estritamente Case-Sensitive por padrão.
Busca Insensível Padrão do sistema na maioria das instalações. Exige o operador específico ILIKE.
Performance Otimizado para LIKE 'Prefixo%' usando índices. Utiliza índices B-tree (Prefixo) ou GIN/GIST para buscas complexas.

2. BETWEEN (Faixa de Valores)

O grande diferencial do BETWEEN é a legibilidade e a limpeza do código.

  • Inclusividade: Ele sempre inclui os "muros" da busca. BETWEEN 10 AND 20 é exatamente valor >= 10 AND valor <= 20.
  • Datas: É a forma mais elegante de filtrar períodos. Ex: BETWEEN '2024-01-01' AND '2024-01-31'.
  • Atenção: Em campos do tipo DATETIME, o limite superior '2024-01-31' é lido como '2024-01-31 00:00:00'. Registros do final do dia podem ser excluídos se você não considerar o horário.

3. IN (Membros de um Conjunto)

O IN atua como um "redutor de múltiplos operadores OR".

  • Cenário Básico: Substitui a repetição cansativa de estado = 'SP' OR estado = 'RJ' OR estado = 'MG' por apenas estado IN ('SP', 'RJ', 'MG').
  • Cenário Avançado: Funciona como uma "ponte" entre tabelas via **Subconsultas**. Você pode perguntar: "Quem são os clientes que compraram o produto X?", e o IN resolve a lista de IDs dinamicamente para você.

Exemplo de Código

Busca por Padrões (LIKE)

Cenário: Busca parcial por nomes de produtos e validação de padrões fixos.

-- [ MySQL ]
-- Busca insensível a maiúsculas (dependendo do collation)
SELECT descricao FROM produto WHERE descricao LIKE 'Gel%';

-- [ PostgreSQL ]
-- O LIKE no Postgres é case-sensitive. Use ILIKE para ignorar a caixa.
SELECT descricao FROM produto WHERE descricao ILIKE 'gel%';

-- [ Padrões Fixos ]
-- Encontra CPFs com padrão exato (usando _ para posições fixas)
SELECT nome FROM vendedores WHERE cpf LIKE '123.______.____-00';

Filtro de Faixas (BETWEEN)

Cenário: Filtragem de preços e períodos de datas.

-- Filtro de preço inclusivo (valor >= 10 AND valor <= 50)
SELECT descricao_produto, valor_unitario 
FROM produto 
WHERE valor_unitario BETWEEN 10.00 AND 50.00;

-- Filtro de Datas: Cuidado com o limite invisível (time 00:00:00)
SELECT id_pedido, data_pedido 
FROM pedidos 
WHERE data_pedido BETWEEN '2024-01-01' AND '2024-03-31';

Pertencimento a Listas (IN)

Cenário: Filtros de múltipla escolha e subconsultas dinâmicas.

-- Substituindo múltiplos ORs
SELECT nome_vendedor FROM vendedores 
WHERE unidade_federativa IN ('SP', 'RJ', 'MG', 'ES');

-- Avançado: Subconsulta (Vendedores sem vendas em maio)
SELECT nome_vendedor FROM vendedores 
WHERE codigo_vendedor NOT IN (
    SELECT codigo_vendedor FROM pedidos 
    WHERE data_pedido BETWEEN '2024-05-01' AND '2024-05-31'
);

Dicas & Lembretes

  • Performance: Prefira LIKE 'Padrão%' a LIKE '%Padrão' para permitir o uso de índices B-tree.
  • Sensibilidade: PostgreSQL é case-sensitive (LIKE vs ILIKE), MySQL depende do collation.
  • Valores Nulos: Predicados como LIKE, IN e BETWEEN não capturam NULL. Use IS NULL explicitamente se necessário.
  • Limites: O BETWEEN sempre inclui os valores das extremidades (inclusivo).

Como aplicar na Prática

  • E-commerce (Busca Inteligente): Implemente uma barra de pesquisa que utiliza LIKE '%termo%' para buscar em títulos e descrições, permitindo que o cliente encontre o produto mesmo sem saber o nome exato.
  • Relatórios Financeiros (Fechamento Mensal): Utilize o BETWEEN para extrair todas as vendas ocorridas entre o primeiro e o último dia do mês, garantindo um relatório preciso para a contabilidade.
  • Sistemas de Filtros (Checkboxes): Quando um usuário seleciona várias categorias (ex: Eletrônicos, Livros, Games) em um filtro lateral, utilize o operador IN para passar essa lista de IDs diretamente para o banco em uma única consulta.
  • Segurança (Blacklist): Bloqueie o acesso de uma lista específica de IPs ou IDs de usuários suspeitos utilizando NOT IN (lista_bloqueada).
