Aula 07: Segurança e Privilégios - Controle de Acesso

Gestão de privilégios e controle de operações no banco de dados.

Teoria & Conceitos

A segurança em Sistemas Gerenciadores de Banco de Dados (SGBD) é uma disciplina essencial que visa proteger os dados contra o uso indevido e garantir que cada usuário acesse apenas as informações estritamente necessárias para suas funções. No contexto da linguagem SQL, esta camada é gerenciada pela DCL (Data Control Language), que trata da Autorização de acesso aos objetos do banco.

1. Autenticação vs. Autorização

Antes de manipular qualquer dado, o SGBD processa duas etapas de segurança distintas:

  • Autenticação: O processo de verificar a identidade do usuário (quem você é). Geralmente envolve um nome de usuário, um host de origem e uma senha.
  • Autorização: Uma vez identificado, o sistema verifica quais privilégios o usuário possui (o que você pode fazer). Esta aula foca na gestão desses privilégios através do dicionário de dados do sistema.

2. Comandos Fundamentais da DCL

Segundo o padrão ANSI SQL e as práticas de mercado, existem três comandos principais para o controle de acesso:

  • GRANT (Conceder): Atribui permissões específicas (como SELECT, INSERT, UPDATE) a um usuário ou papel.
  • REVOKE (Revogar): Remove privilégios anteriormente concedidos, impedindo que o usuário realize certas operações, mas mantendo o acesso que não foi revogado.
  • DENY (Negar): Comando comum em sistemas como SQL Server para proibir explicitamente um acesso. No PostgreSQL e MySQL, a negação é geralmente implícita (se não foi dado o GRANT, o acesso é negado por padrão).

3. Granularidade e Níveis de Acesso

A segurança deve ser aplicada em diferentes níveis de "profundidade" para garantir o Princípio do Privilégio Mínimo:

  • Global (Servidor): Privilégios que afetam todo o servidor de banco de dados (ex: criar novos bancos ou gerenciar outros usuários).
  • Database/Esquema: Permissões para agir dentro de um banco de dados específico.
  • Tabela/Visão: Controle sobre objetos individuais. O analista pode permitir que um usuário leia uma tabela, mas não a altere.
  • Coluna: Nível avançado onde se permite o acesso a campos específicos de uma tabela (ex: um vendedor pode ver o nome do produto, mas não o preço de custo).
  • Rotina (Procedures/Functions): Permissão para executar lógicas de negócio programadas no servidor.

4. Diferenças de Implementação: MySQL vs. PostgreSQL

Embora ambos sigam o padrão SQL para autorização, a arquitetura interna de segurança varia:

Conceito MySQL PostgreSQL
Identidade Baseada em User + Host (ex: 'aluno'@'localhost'). O local de onde o usuário conecta é parte da sua identidade. Baseada em Roles (Papéis). Uma Role pode ser um usuário individual ou um grupo que contém outros usuários.
Armazenamento Privilégios globais são armazenados na tabela de sistema mysql.user. Privilégios são gerenciados via catálogos como pg_authid e pg_roles.
Acesso a Objetos Dar privilégio em um database permite acesso imediato às tabelas. Exige o privilégio de USAGE no SCHEMA (esquema) antes que o usuário possa acessar qualquer tabela dentro dele.
Propriedade O criador da tabela é o Owner e possui controle total sobre ela por padrão. Segue o rigor do Owner, mas permite a transferência de posse de forma mais granular através de ALTER OWNER.

5. Integridade e Segurança

A segurança de acesso é um complemento às Restrições de Integridade. Enquanto as chaves primárias e estrangeiras garantem que os dados sejam logicamente corretos, os privilégios garantem que apenas as pessoas certas possam inserir ou modificar esses dados, mantendo a consistência do banco de dados a longo prazo.

Exemplo de Código

Abaixo, comparamos a sintaxe para as principais operações de segurança.

1. Criação de Usuários

-- [ MySQL ]
CREATE USER 'vendedor_sp'@'localhost' IDENTIFIED BY 'Senha@123';
CREATE USER 'admin_fin'@'%' IDENTIFIED WITH 'sha256_password' BY 'Forte#2024';

-- [ PostgreSQL ]
CREATE USER vendedor_sp WITH PASSWORD 'Senha@123';
-- Em PG, usuários e roles são o mesmo objeto, mas 'user' implica LOGIN
CREATE ROLE auditor WITH LOGIN PASSWORD 'Auditoria!99';

2. Gestão de Privilégios

-- [ MySQL ]
-- Global
GRANT ALL PRIVILEGES ON *.* TO 'super_user'@'localhost';
-- Database/Tabela
GRANT SELECT, INSERT ON vendas.* TO 'gerente'@'%';
-- Coluna
GRANT SELECT (nome, cargo) ON vendas.EMPREGADO TO 'recepcionista'@'localhost';

-- [ PostgreSQL ]
-- Global (Superuser)
ALTER ROLE super_user WITH SUPERUSER;
-- Database
GRANT CONNECT ON DATABASE vendas TO gerente;
-- Schema/Tabela (PostgreSQL requer permissão no Schema)
GRANT USAGE ON SCHEMA public TO gerente;
GRANT SELECT, INSERT ON ALL TABLES IN SCHEMA public TO gerente;

3. Manutenção e Auditoria

-- [ MySQL ]
REVOKE DELETE ON vendas.* FROM 'gerente'@'%';
SHOW GRANTS FOR 'gerente'@'%';
DROP USER 'vendedor_sp'@'localhost';

-- [ PostgreSQL ]
REVOKE DELETE ON ALL TABLES IN SCHEMA public FROM gerente;
-- Auditoria via catálogo ou psql
\du              -- Lista roles no psql
SELECT * FROM information_schema.role_table_grants WHERE grantee = 'gerente';
DROP ROLE vendedor_sp;

Dicas & Lembretes

  • MySQL Host: 'aluno'@'localhost' é diferente de 'aluno'@'%'.
  • PostgreSQL Host: O controle de acesso por IP e método de conexão é feito no arquivo externo pg_hba.conf.
  • Princípio do Privilégio Mínimo: Nunca conceda ALL PRIVILEGES ou SUPERUSER se apenas SELECT for necessário.
  • Esquemas no PG: Lembre-se que no PostgreSQL, dar grant em uma tabela exige também permissão de USAGE no SCHEMA correspondente.

Como aplicar na Prática

  • Auditoria: Utilize SELECT User, Host, plugin FROM mysql.user; para listar todos os usuários.
  • Fraquezas: Procure por usuários sem senha ou que utilizam plug-ins depreciados.
  • Segregação de Funções: Crie usuários distintos para desenvolvedor, analista e sistema de backup.
