Aula 01: Estrutura Física do SGBD

Entendendo como os dados são realmente armazenados nos dispositivos físicos e a arquitetura de arquivos do SGBD.

Teoria & Conceitos

O Modelo Físico (ou Nível Interno, na arquitetura ANSI/SPARC) descreve como os dados são realmente armazenados nos dispositivos físicos. Enquanto os modelos conceitual e lógico omitem detalhes técnicos, o modelo físico foca em nomes de arquivos, métodos de endereçamento e caminhos de acesso.

No Sistema Gerenciador de Banco de Dados (SGBD), a estrutura física é organizada em:

  • Arquivo de Dados Primário (.MDF): O ponto de entrada do banco de dados. Contém as informações de inicialização e as tabelas do sistema.
  • Arquivos de Dados Secundários (.NDF): Utilizados para expandir o armazenamento quando o arquivo primário atinge o limite ou para distribuir os dados em múltiplos discos físicos (devices), melhorando a performance de I/O.
  • Log de Transações (.LDF): Um registro sequencial de todas as operações de modificação. É vital para a recuperação do banco em casos de falhas, permitindo operações de commit e rollback.

Exemplo de Código

Abaixo, o comando DDL para criar um banco de dados distribuído em múltiplos arquivos físicos:

-- Criação do banco de dados com arquivos MDF, NDF e LDF
CREATE DATABASE VendaMais
ON PRIMARY (
    NAME = 'VendaMais_Prim',
    FILENAME = 'C:\SQLData\VendaMais.mdf',
    SIZE = 50MB,
    MAXSIZE = 500MB,
    FILEGROWTH = 10MB
),
(
    NAME = 'VendaMais_Secund',
    FILENAME = 'D:\SQLData\VendaMais_2.ndf', -- Arquivo Secundário em outro disco
    SIZE = 50MB,
    FILEGROWTH = 10MB
)
LOG ON (
    NAME = 'VendaMais_Log',
    FILENAME = 'E:\SQLLogs\VendaMais.ldf', -- Log em disco isolado
    SIZE = 25MB,
    FILEGROWTH = 5MB
);

Dicas & Lembretes

  • Isolamento de Logs: Sempre armazene o arquivo .LDF em um disco físico diferente dos arquivos de dados (.MDF/.NDF) para garantir que uma falha de hardware em um disco não impossibilite a recuperação dos dados.
  • Independência Física: Um bom projeto físico permite que você altere o local dos arquivos no SGBD sem afetar a aplicação Java ou Web que consome os dados.
  • Boas Práticas de NDF: Utilize arquivos secundários para tabelas muito grandes ou índices, distribuindo a carga de leitura entre os discos.

Como aplicar na Prática

  • Planeje a volumetria de dados baseada no Minimundo (quantas vendas por dia?).
  • Configure o Autogrow (FILEGROWTH) com valores fixos em vez de porcentagens para evitar fragmentação excessiva do disco.
  • Crie o script de banco de dados garantindo que os diretórios físicos existam no servidor.
